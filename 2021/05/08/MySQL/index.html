多版本并发控制，用来保证事务的隔离级别，方便回滚

索引

索引类型

主键索引

表的主键列对应的索引就是主键索引，并且一张表只能有一个主键，主键不能为null，不能重复。在InnoDB中，如果一张表里没有主键，就会寻找是否存在唯一索引作为默认的主键，如果没有唯一索引存在，就会自动创建一个6Byte的自增主键。

索引优化

Explain

Explain的各个字段的含义：

  • id: 查询的顺序，id越大最先执行，id相等排序靠前先执行

  • select_type:

    1. simple: 简单查询，查询不包含子查询或UNION
    2. primary: 复杂查询中的最外层的查询
    3. subquery: 包含在子查询中的查询（不在from字句里）
    4. derived: 包含在from字句中的子查询。MySQL会将结果存放在一个临时表中，也称为派生表
    5. union: 在union中的第二个和随后的select查询

  • table: 查询的表

  • partitions: 分区（一般用不到，都是分库分表）

  • type: 这一列表示关联类型或访问类型，即MySQL决定如何查找表中的行，查找数据行记录的大概范围。依次从最优到最差分别为：system > const > eq_ref > range > index > ALL。一般来说保证达到range级别，最好到ref。

    NULL：MySQL能在优化阶段分解查询语句，在执行阶段就不用访问表或者索引。比如在索引中选最小值，可以直接走索引来完成。

    const: 常量查询，一般是直接定位一条数据

    system: 查一张表时，这张表只有一条记录，算时const的一种特例

    eq_ref: 如果有表关联（join连接之类），查询时用的是这个表的primary key或unique key，且最多只会返回一条符合条件的记录，那就是eq_ref。简单的select查询不会出现这种type。

    ref: 相比eq_ref，不使用唯一索引，而是使用普通索引，或者唯一索引的部分前缀（联合索引），索引要和某个值比较，可能会找到多个符合条件的行。

    range: 范围扫描，出现在in(), between >, <, =等操作。

    index: 覆盖索引的概念，就是从非主键索引的叶子节点从头开始扫描到尾。

    ALL: 全表扫描，扫描聚簇索引的所有叶子节点。

  • possible_keys:

  • key:

  • key_len:

  • ref:

  • rows:

  • filtered:

  • Extra:

索引优化情况

  1. 全值匹配:
    联合索引的话尽量让索引的长度越高越好，也就是索引用的越多越好。
  2. 最左前缀法则：
  3. 不要再索引列上做任何操作（计算、函数、（自动或手动）类型转换），会导致索引失效
  4. 存储引擎不能使用索引中范围条件右边的列。比如只走了联合索引前两个索引，没有走第三个索引，因为第三个索引是在这个范围内是无序的，所以走不了索引。
  5. 尽量使用覆盖索引（只访问索引的查询（索引包含查询列）），减少select * 语句
  6. MySQL在使用不等于（！=或者 < 加上 > ）的时候无法使用索引会导致全表（因为结果集太大了）
  7. is null, is not null 一般情况下也无法使用索引
  8. like以通配符开头（’$abc…’）MySQL索引失效会变成全表扫描（通配符开头会破坏有序性，因为比较顺序一般都是根据字段的前几个字符来排序的）
  9. 字符串不加单引号索引失效
  10. 少用or或in，用它查询时，MySQL不一定使用索引，MySQL内部优化器会根据检索比例、表大小等多个因素整体评估是否使用索引。
  11. 范围查询优化
    给年龄添加单值索引。可以把范围拆成几个段。

    模糊查询

模糊查询如果用不了索引，可以使用覆盖索引，这样就能用索引。

索引下推

在索引遍历过程中，虽然可能第一个索引导致后面的索引无序，但是可以对后面的索引进行一个判断，过滤掉不符合条件的记录再回表，这样可以减少回表次数。

排序

分为索引排序和文件排序，文件排序效率低

索引设计原则

  1. 代码先行，索引后上
  2. 联合索引尽量覆盖条件
  3. 不要再小基数字段上建立索引
  4. 长字符串可以使用前缀索引
  5. where和order by冲突时优先where
  6. 尽量避免单值索引，多用联合索引

锁

悲观锁

乐观锁

表锁

每次锁一张表，开销比较小，加锁快，不会出现死锁，锁粒度大，发生冲突概率大，并发度低，一般用在整表迁移

行锁

每次操作锁一行数据，开销大，加锁慢，会出现死锁，锁粒度最小，发生锁冲突的概率最低，并发程度最高。

间隙锁

间隙锁，锁的是两个值之间的间隙，意思是把这个间隙给锁住。一定程度上可以在可重复读的级别下解决幻读的问题。只能在可重复读下生效。

临键锁

是行锁和间隙锁的组合。

无索引行锁会升级为表锁

锁主要是加在索引上，如果对非索引字段更新，行锁可能会变成表锁

锁优化建议

  • 尽可能让所有数据检索都通过索引来完成，避免无索引行锁升级为表锁
  • 合理设计索引，尽量缩小锁的范围
  • 尽可能减少检索条件范围，避免间隙锁
  • 尽量控制事务大小，减少锁定资源量和时间长度，设计事务加锁的sql尽量放在事务的最后执行
  • 尽可能低级别事务隔离
